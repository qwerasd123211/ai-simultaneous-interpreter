# LINGUA 项目 Demo 视频讲解脚本

建议视频时长：3 到 5 分钟。

## 0. 开场

大家好，我的项目叫 LINGUA，是一个 AI 同声传译助手。

这个项目解决的问题是：当我们观看英语演讲、技术分享、国际会议或者网课时，语速快、术语多，很容易跟不上内容节奏。LINGUA 会把浏览器里正在播放的英文音频实时识别并翻译成中文，用主页面字幕和悬浮字幕窗口展示出来，帮助用户更快理解内容。

## 1. 展示项目入口

现在我打开项目网页。

可以看到页面不是普通介绍页，而是一个同声传译工作台。左侧是翻译控制和术语表，中间是实时字幕区域，右侧是延迟指标和历史记录。

这里的设计目标是让用户一进来就能操作，不需要阅读很长的说明。

## 2. 讲核心使用流程

接下来我点击“开始翻译”。

浏览器会弹出屏幕共享选择器。我选择正在播放英文视频的标签页，并且勾选“共享音频”。这一步很关键，因为系统要拿到视频里的声音，而不是麦克风声音。

选择完成后，系统会打开悬浮字幕窗口。这样用户可以一边看英文视频，一边在旁边看到中文字幕。

## 3. 展示实时字幕效果

现在我播放一段英文朗读。

可以看到，英文原文和中文翻译会持续出现在字幕窗口里。主页面的实时字幕区也同步更新。

右侧可以看到 ASR 延迟、翻译延迟、总延迟和修正状态。这样评审可以直观看到系统不是离线批处理，而是在实时处理音频流。

## 4. 展示术语表

接下来展示术语表能力。

比如技术分享里经常出现 Kubernetes、API、React 这样的词。如果普通翻译把这些词误译，会影响理解。

我可以在术语表里写：

```text
Kubernetes => Kubernetes
API => API
React => React
```

系统会把这些术语传给后端翻译提示词，优先保留或按指定方式翻译，从而提升技术内容的准确性。

## 5. 展示自动修正能力

题目要求系统需要具备修正能力，能够自动纠正之前识别或翻译的错误。

LINGUA 使用的是流式 ASR，中间结果会先显示出来。如果后续 ASR 判断前面的词识别错了，会返回动态修正片段。

我在后端做了片段级合并：系统会根据 ASR 返回的片段编号和修正范围，只替换出错的片段，而不是整段覆盖。这样用户看到的是字幕原地修正。

页面里的“演示模式”可以快速展示这个效果。比如一开始把 Kubernetes 识别错，后面系统会自动修正为 Kubernetes，并且修正状态会变化。

## 6. 讲快语速优化

实际测试中我发现，如果英语语速比较快，字幕容易反应慢，甚至漏句。

这里我做了两点优化：

第一，修复了 ASR 动态修正合并逻辑。之前后面的修正片段可能覆盖前面的句子，造成漏句。现在按片段编号保存，再按顺序拼接，快语速下也能保留前文。

第二，降低了中间结果翻译的等待时间，并缩短静音断句阈值，让字幕更快刷新。

所以这个项目不是只做了一个界面，而是在真实同传场景里针对延迟和漏句做了优化。

## 7. 讲技术架构

简单介绍一下技术架构。

前端使用 HTML、CSS 和 JavaScript。用户点击开始后，前端通过浏览器的 `getDisplayMedia` 获取共享标签页音频，再用 `AudioContext` 把音频转成 PCM，并降采样到 16kHz。

后端是 Node.js、Express 和 WebSocket。前端每约 100ms 把音频帧发送给后端。

后端连接讯飞流式 ASR 获取英文识别结果，再调用 DeepSeek 翻译成中文，最后通过 WebSocket 推回前端。

字幕窗口、主页面字幕和历史记录都会同步更新。

## 8. 展示健康检查和部署

项目还提供了 `/health` 健康检查接口，可以看到服务状态，以及 DeepSeek 和讯飞密钥是否配置。

后端适合部署在 Railway，因为它需要长期运行 WebSocket 服务。前端也可以部署到 Vercel，并通过配置连接 Railway 后端。

## 9. 总结

总结一下，LINGUA 对应题目“AI 同声传译助手”的要求，已经实现了：

第一，单向英文音频流实时识别。

第二，英文到中文的实时翻译。

第三，主页面和悬浮窗字幕展示。

第四，ASR 和翻译修正能力。

第五，针对快语速场景的连续性和延迟优化。

第六，术语表、延迟指标、历史记录和字幕导出等辅助能力。

我的目标是让用户在看英语视频或技术分享时，不需要打断观看节奏，也能持续跟上内容。

以上就是我的项目演示，谢谢大家。
